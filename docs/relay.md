# Relay

The relay is optional and out-of-band. PostgreSQL enforcement does not require it.

## Database Side

For warn/block-worthy events, the extension emits:

```sql
NOTIFY pg_lockstep_events, '<json payload>';
```

The JSON payload is the same event shape stored in `lockstep.audit_log.event`.

## Local Process

```sh
python relay/pg_lockstep_relay.py --dsn postgresql://postgres:postgres@localhost:5432/postgres
```

The relay prints each event as JSON.

## Slack

If both variables are set, the relay posts to Slack:

```sh
export PG_LOCKSTEP_SLACK_BOT_TOKEN=xoxb-...
export PG_LOCKSTEP_SLACK_CHANNEL=C0123456789
```

Slack failures are logged and ignored. The relay has basic in-memory deduplication by `event_id` and rate limiting.

## Test Slack Delivery

```sh
python relay/pg_lockstep_relay.py --test-slack
```

This sends a synthetic Slack message using `PG_LOCKSTEP_SLACK_BOT_TOKEN` and `PG_LOCKSTEP_SLACK_CHANNEL`.

## Blocked Statement Caveat

PostgreSQL delivers `NOTIFY` only after transaction commit. A blocked command raises an exception, so PostgreSQL rolls back the audit insert and `NOTIFY` generated by that statement. The relay receives committed warn/observe notifications, but not notifications from statements blocked by the same transaction. The blocked event JSON remains available in the SQL exception `DETAIL`.

## Blocked Alerts With dblink

To receive alerts for blocked commands from the SQL/event-trigger release, enable the optional local `dblink` path. This sends `NOTIFY` from a separate local PostgreSQL transaction before the original command is blocked.

```sql
CREATE EXTENSION IF NOT EXISTS dblink;
SELECT lockstep.configure_blocked_alerts(
  true,
  'dbname=topology_db',
  true
);
```

The third argument enables a trusted-channel approval token in the event payload. The relay includes that token in Slack. Only enable this for protected Slack channels, because anyone who can read the token can retry the matching command until the token expires or is consumed.
